# プロダクト概要

## プロジェクト概要

- **対象**: 日本の大手小売店内プロジェクト。利用者は海外の販社（20か国以上）担当者
- **課題**: 現状メールベースの問い合わせ・申請対応が非構造化で、日本のヘルプデスク側の分類作業が負担になっている
- **目的**: 問い合わせ・申請を統一フォーム化し、選択式項目で自動分類。加えて情報共有・リンク集を1つのポータルに集約する

## 開発フェーズ方針

| フェーズ | 概要 | 状態 |
|---|---|---|
| フェーズ1 | Next.jsによるフロントエンドのみのモックアップ作成 | 完了 |
| フェーズ3 | DB・バックエンドAPI実装（PostgreSQL・Prisma・Auth.js）。モックAPIを実APIに差し替え | 完了（PR #32でmainにマージ済み。詳細は`tech.md`） |
| フェーズ2（現在・未着手） | ヘルプデスク担当者へのヒアリング結果を反映し、分類軸（`category`等の仮enum）・画面を修正 | 未実施。フェーズ3が先行して完了したため、現在の`category`等はヒアリング未反映の仮値のまま本番DBに存在する（`structure.md`参照） |

> 番号はフェーズ3がフェーズ2より先に完了した実態の経緯を示す（当初想定の順序と実施順が入れ替わっている）。

## 主要機能

申請者側（海外販社担当者向け）:

| 機能 | 概要 |
|---|---|
| ダッシュボード | お知らせ・問い合わせステータス概要、各機能へのナビゲーションカード |
| 問い合わせ・申請フォーム | 選択式項目＋自由記述。分類・緊急度・地域などを選択式で入力し分類作業を削減 |
| 問い合わせ一覧 | 自分（自社）が出した問い合わせの状況確認、検索・絞り込み、追加メッセージ送信 |
| お知らせ・情報共有 | ヘルプデスクからの周知事項。対応要否・対応期限・既読/実施済みトラッキング |
| ドキュメント | ヘルプデスクが配布する資料の閲覧（ファイル添付・Google Docs/Sheets/Slides共有リンク） |
| リンク集 | カテゴリ別の外部/内部サイトリンク |
| FAQ | よくある質問 |

ヘルプデスク側（日本本社担当者向け）:

| 機能 | 概要 |
|---|---|
| 問い合わせ管理（申請管理） | 全社の問い合わせに対する対応中フラグ・返信（テンプレート含む）・対応履歴・代理登録 |
| お知らせ管理 | お知らせの下書き/公開・国別配信対象設定・リマインド送信 |
| ドキュメント管理 | 配布資料の登録・編集（ファイル添付・Google共有リンク登録） |
| リンク集・FAQ管理 | 申請者側に表示するリンク集・FAQの登録・編集 |
| テンプレート管理 | 返信テンプレートのカテゴリ・名称・本文管理 |

共通: 認証（Auth.js、Credentials Provider、ロール別セッション）

## 機能実装推奨順序（初期実装時の順序・完了済み）

1. `dashboard`（土台となるレイアウト・ナビゲーション）
2. `inquiry-form`（最重要機能。フォーム項目・分類軸を型に落とし込む）
3. `inquiry-list`（自社が出した問い合わせの一覧・状況確認。`inquiry-form`が定義した型・モックAPIを参照する）
4. `announcements`（お知らせ・情報共有）
5. `links-page`（リンク集）
6. `faq`（FAQ）

上記は全て実装済み。以降、`documents`・各ヘルプデスク管理画面・`backend-db-foundation`・`brand-alignment`等が追加され、`.kiro/specs/`配下の各specとして管理されている。次の開発優先度は都度検討する（現時点の候補は`.kiro/steering/`の分析結果を参照）。
